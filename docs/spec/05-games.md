# 05 — Games UI (ANDGAME)

Client for the backend game platform (`roadtrip-backend/docs/spec/08-games.md`). Game actions
are **online-only** (server arbitrates turns); the UI must degrade gracefully offline.

Screens: Lobby (open games, my games, challenge inbox) → Board (per game type) →
Replay (finished games, play/pause/step) and Spectate (live view of someone else's game).

## Requirements

| ID | Requirement | Verify |
|----|-------------|--------|
| ANDGAME-001 | The lobby lists open games to join, the profile's active games (with turn indicator), and incoming challenges; data reduces from `GET /api/games` + the event feed. | auto |
| ANDGAME-002 | Creating a game offers mode (open/challenge with profile picker) and game-specific options (hangman: word/phrase entry with dictionary toggle and live cap validation mirroring GAME-013 rules). | auto |
| ANDGAME-003 | Boards render server state (engine `view`) and submit moves via `POST /api/games/{id}/moves`; a rejected move (400/409) restores the pre-move UI state with the server's reason shown. | auto |
| ANDGAME-004 | All five games have playable boards: chess, checkers, tic-tac-toe, ultimate tic-tac-toe (with dictated-sub-board highlighting), hangman (masked phrase with visible word boundaries, gallows progress). | manual |
| ANDGAME-005 | While a game screen is open, the client long-polls the game event stream so the opponent's move appears without manual refresh (~1s VPN latency tolerated). | auto |
| ANDGAME-006 | Replay mode steps through the recorded move stream with play/pause/step controls, rebuilding board states locally from events (client-side replay determinism against the core reducers). | auto |
| ANDGAME-007 | Spectate mode is replay pinned to live tail: it follows the stream as moves arrive; any profile can spectate any active game. | auto |
| ANDGAME-008 | Offline: game actions are disabled with an explanatory banner; finished-game replays cached locally still work. | auto |
| ANDGAME-018 | The board's end-of-game control is role- and type-aware (mirrors backend GAME-015): for hangman, the creator/setter (host) sees an "End game" control that abandons the game via the resign endpoint, and the guesser sees no end/resign control; for all other game types, any active-game participant still sees "Resign". Spectators and non-active games show no control. | auto |
| ANDGAME-015 | Hangman turn belongs to the single guesser (the joining opponent), never the word-setter/creator: the reducer assigns the turn to the guesser on `game.joined` and keeps it there on every `game.move` — no creator↔opponent alternation — mirroring the backend rule that the setter never takes a turn (`roadtrip-backend` GAME-013; server `Game.turn`). | auto |
| ANDGAME-016 | The hangman board is built from the backend's viewer-aware engine `view` (`display`, `guessed`, `wrong`, `max_wrong`) returned by `GET /api/games/{id}`, not re-folded from the events feed: the word is redacted to `•••` for ongoing games (`roadtrip-backend` GAME-014), so local folding would miscount every guess and garble the mask. The live long-poll re-fetches the view when new game events arrive. | auto |
| ANDGAME-017 | The hangman status line reflects the single guesser's ongoing turn (guesser sees "your turn"; the setter waits on the guesser) and the win/loss end state, and never falls through to the generic "waiting for the other player" — there is no second guesser. | auto |
| ANDGAME-009 | Every lobby row identifies who started the game: rows created by others read "<Name>'s <GameType>", resolved from the profiles cache with a neutral fallback ("Someone") when the creator isn't cached; the profile's own games attribute the opponent ("<GameType> vs <Name>") or "Your <GameType>" when solo/unknown. Game type stays identifiable. Resolution lives in a testable seam (`GameLobbyLabeler`). | auto |
| ANDGAME-010 | The lobby offers both pull-to-refresh and an explicit header refresh button, sharing one reload action (`LobbyRefresher`) that requests a foreground sync and refetches the games cache; the indicator shows while reloading and dismisses on completion, and offline skips the network reload rather than spinning indefinitely. | auto |
| ANDGAME-011 | The lobby list reserves a bottom content inset large enough to clear the floating "new game" button (`GamesLobbyLayout.FAB_CLEARANCE_DP`) so the last row's action stays tappable in every section, not just finished games. | auto |
| ANDGAME-012 | Grid boards (chess, checkers, tic-tac-toe, ultimate) are bounded to the smaller of the available width/height so the full board fits the viewport without scrolling on tablet (`WindowSizeClass` medium/expanded) in portrait and landscape; piece glyphs scale to a fixed fraction (~70%) of their square rather than a fixed point size. The board is kept out of any vertical scroll region. Sizing math lives in `BoardMetrics` (pure/JVM). | auto |
| ANDGAME-013 | Piece rendering uses shared pure mappings (`BoardPieces`): chess draws the SOLID Unicode glyphs for BOTH sides, distinguished by ink — white pieces render as a white fill with a dark outline so they read on the light square, black pieces as black; checkers renders red vs black (never white), both legible on their squares. | auto |
| ANDGAME-014 | When a tic-tac-toe or ultimate game is won, core exposes the winning line cell indices (`tttWinningLine`, plus the ultimate macro line and per-captured-sub-board lines) so the view can highlight exactly those tiles; a finished game (win or draw) reads as clearly over. | auto |
| ANDGAME-019 | Hangman lobby rows append the server-redacted masked progress (`Game.hangman_display`, `roadtrip-backend` GAME-001) to the creator/opponent title, e.g. "Derek's Hangman — E_E___NT", truncated at a fixed character cap (`GameLobbyLabeler.HANGMAN_MASK_CAP`) with a trailing ellipsis when longer. Non-hangman games and hangman games without a display (open/unstarted) keep the plain title unchanged. The mask is used verbatim — the title never reveals letters beyond the already-redacted server mask. | auto |
| ANDGAME-021 | The lobby is split into two tabs below the "Games" header — **Active** and **Finished** — driven by the pure `LobbyTabs` seam. Active shows (in order) the License-plate-bingo family entry pinned at the top, then My games, Challenges for you, Open games, and Watch live; Finished shows the finished-game replay archive. A game is bucketed to Finished iff `status == FINISHED` (`LobbyTabs.tabFor`); every other game plus the static bingo entry is Active. The screen always opens on Active (`LobbyTabs.DEFAULT`): the selected tab is held in plain local composable state (no `rememberSaveable`, no persistence), so returning to Games or a process restart resets to Active. Pull-to-refresh, the header refresh button, the offline banner, and the "new game" FAB (with the ANDGAME-011 `FAB_CLEARANCE_DP` bottom inset) stay shared across both tabs and apply to whichever list is showing. Each tab keeps its own empty state — Finished shows "No finished games yet" when empty; Active keeps its per-section placeholders. | auto |
| ANDGAME-022 | The board carries a per-profile, per-game-type "Confirm move before making it" toggle (chess, checkers, tic-tac-toe, ultimate, hangman), settable mid-game on the board screen itself (never in Settings). When on, a tapped move is staged and `POST /api/games/{id}/moves` only fires after an explicit **Confirm**; **Cancel** discards the staged move and restores the pre-move board state (same restore idiom as ANDGAME-003). When off, moves submit immediately as before. The value is scoped to a single `(profileId, gameType)` pair and never affects the profile's other game types or other players. The initial default for any never-set pair is on (`ConfirmMovePreference.resolve(stored) = stored ?: true`); each set is sticky and last-write-wins, persisted locally (`ConfirmMoveStore`, key `confirm_move:<profileId>:<gameType>`) across app restarts, client-side only (no backend change). The default-resolution and the stage→confirm/cancel decision live in the pure JVM `ConfirmMovePreference` and `MoveConfirmGate` seams. | auto |
| ANDGAME-023 | Resigning or host-ending a game is destructive and irreversible (it hits `POST /api/games/{id}/resign` and ends the game for both players), so the board's end-of-game control (ANDGAME-018) ALWAYS requires an explicit confirmation before the resign request is issued — unlike the ANDGAME-022 move-confirm toggle, this is never a setting and never toggleable. Tapping the control (the plain "Resign", or hangman's host "End game") opens a confirmation dialog ("Resign this game? This can't be undone." / "End game?" for the hangman host, with a destructive confirm and a Cancel); the resign endpoint fires ONLY on explicit confirm, and Cancel dismisses the dialog leaving the game untouched (no request sent). Both the `EndControl.RESIGN` and `EndControl.END_GAME` paths go through the confirmation; `EndControl.NONE` (spectators, non-active games) has no control and nothing to confirm. The request→pending, confirm→invoke-resign-exactly-once, cancel→no-op decision lives in the pure JVM `ResignConfirmation` seam. Client-side only (no backend change). | auto |
| ANDGAME-020 | Tic-tac-toe and ultimate boards show a player-identity legend mapping each mark to a player name — `X → Game.createdBy` (first player), `O → Game.opponentId` — resolved from the profiles cache with the neutral "Someone" fallback (`GameLobbyLabeler.UNKNOWN_PLAYER`) when a player isn't cached or is absent. The on-turn player's row (active game && `Game.turn` equals that player's id) is boxed with the identical `3.dp` `colorScheme.primary` border the board draws for the active/selected square. The legend sits below the board on phone and beside it on tablet/wide, switched on the same width threshold as ANDGAME-012 (`PlayerLegend.placeBeside`, medium/expanded ≈ `BESIDE_MIN_WIDTH_DP` = 600dp). Non-mark games (chess, checkers, hangman) show no legend. Mapping, turn, and placement live in the pure `PlayerLegend` seam. | auto |
| ANDGAME-024 | Every board highlights the SINGLE most-recent move so a player sees what just changed. Only that one move is ever highlighted and it RESETS on each new move (no accumulating trail); there is no highlight before the first move. Treatment is per game type and is always a FILL/SHADE (never a border), so it stays visually distinct from the ANDGAME-020/ANDGAME-012 `3.dp` `colorScheme.primary` selected/on-turn border and from the ANDGAME-004 dictated-sub-board highlight: mark games (tic-tac-toe, ultimate) shade the BACKGROUND of the last-placed cell (`LastMoveShade.MARK_FILL`); piece games (chess, checkers) grey the ORIGIN square darker than other squares (`PIECE_ORIGIN`) and tint the DESTINATION square a dim accent (`PIECE_DESTINATION`), and checkers additionally marks each captured/jumped square (`PIECE_CAPTURE`); hangman indicates the most recent guessed letter. The highlight updates automatically when the opponent's move arrives over the ANDGAME-005 long-poll. In replay/spectate it follows the CURRENT cursor (`ReplaySession.index`), not always the final move — the last move is the one at `index - 1`, so `index <= 0` yields no highlight and advancing/scrubbing the cursor moves the single highlight. A finished game still shows the final-move highlight alongside the ANDGAME-014 win-line highlight. Last-move resolution and the shade-role mapping live in the pure JVM `LastMove` / `LastMoveHighlight` / `LastMoveShade` seam. | auto |
